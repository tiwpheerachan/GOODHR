## 1. ภาพรวมระบบ

ระบบประเมิน KPI ของ GOODHR ทำหน้าที่:

- 📝 **บันทึกผลประเมิน** ของพนักงานรายเดือน
- 💰 **คำนวณเงินรางวัล** เข้า payroll อัตโนมัติ
- 🔄 **รองรับ flow** หัวหน้าตรง + หัวหน้าระดับสูง + ผู้ประเมินเพิ่มเติม
- ✅ **HR ตรวจสอบ** ก่อนแจ้งพนักงาน

### 🌐 อิง "ลูกน้องจริง" — ข้ามบริษัท / ข้ามแผนกได้

ระบบใช้ความสัมพันธ์จาก `employee_manager_history` เป็นหลัก **โดยไม่กรองตามบริษัทหรือแผนก**

ตัวอย่าง: A อยู่บริษัท X, ลูกน้อง C อยู่บริษัท Y → A ยังประเมิน C ได้ปกติ
ตัวอย่าง: B อยู่ฝ่ายขาย, ลูกน้อง D อยู่ฝ่ายคลัง → B ยังประเมิน D ได้ปกติ

> 💡 ถ้าต้องการแบ่งสิทธิ์ตามบริษัท/แผนก → ใช้ตอนตั้ง "หัวหน้า" ใน `employee_manager_history` ให้ตรงตามที่ต้องการ ระบบ KPI จะอิงตามนั้นโดยตรง

### Flow การทำงานหลัก

```
   ┌─────────────┐    ┌─────────────┐    ┌──────────────┐
   │  หัวหน้า     │───▶│   HR ตรวจ   │───▶│  แจ้งพนักงาน  │
   │  ประเมิน    │     │   อนุมัติ    │     │  + payroll   │
   └─────────────┘    └─────────────┘    └──────────────┘
        ⤴
   (กรณีถูก reject — หัวหน้าแก้แล้วส่งใหม่)
```

### สถานะของฟอร์ม (`status`)

| สถานะ | ความหมาย | ใครเห็น |
|---|---|---|
| `draft` | บันทึกร่าง ยังไม่ส่ง | หัวหน้าเอง |
| `submitted` | ส่งแล้ว รอ HR ตรวจ | หัวหน้า + HR |
| `approved` | HR อนุมัติแล้ว | ทุกคน + payroll ใช้ได้ |
| `rejected` | HR ส่งคืนให้แก้ | หัวหน้า (รอแก้ + ส่งใหม่) |
| `acknowledged` | พนักงานรับทราบแล้ว | ทุกคน |

## 2. บทบาทผู้ใช้ (Roles)

ระบบมี 4 บทบาทหลัก:

| Role | ดูในระบบ | ทำอะไรได้ |
|---|---|---|
| `employee` | พนักงานทั่วไป | ดูผลประเมินของตัวเอง (เฉพาะที่ approved) |
| `manager` | หัวหน้าทีม | ประเมินลูกน้องตัวเอง + ลูกน้องในสาย + ที่ assign เพิ่ม |
| `hr_admin` | HR ของบริษัท | อนุมัติ/ส่งคืน/แก้ทุกฟอร์มในบริษัท |
| `super_admin` | ผู้ดูแลทั้งระบบ | เห็นทุกบริษัท + ทำได้ทุกอย่าง |

> 💡 **บทบาทเพิ่มเติม** ที่ไม่ใช่ role แต่เป็น "ความสัมพันธ์ในฟอร์ม":
> - `direct_manager` — หัวหน้าตรง (default)
> - `skip_level` — หัวหน้าระดับสูง (1 ขึ้นไป)
> - `additional` — ผู้ประเมินเพิ่มเติม (assign โดย Admin)

## 3. ประเภทการประเมิน 3 รูปแบบ

หัวหน้าเลือกประเภทได้ตามรายบุคคล รายเดือน (ไม่ผูกตายตัว)

### 🔹 Mode A: มาตรฐาน (`standard`)

**เหมาะกับ:** พนักงานที่มีฐาน KPI กำหนดไว้ใน `kpi_bonus_settings`

**Flow:**
```
หัวข้อ → คะแนน → น้ำหนัก → เกรด → ฐาน KPI × ตัวคูณ → เงินรางวัล
```

**สูตรเงิน:**
- เกรด A: ฐาน × **1.2**
- เกรด B: ฐาน × **1.0**
- เกรด C: ฐาน × **0.8**
- เกรด D: × 0 (ไม่ได้)

**ตัวอย่าง:** ฐาน 5,000 บาท + เกรด A → ได้ 6,000 บาท

**Optional:** เพิ่ม "ค่าผลงานพิเศษ" (bonus) ได้

### 💰 Mode B: ใส่เงินเอง (`money_only`)

**เหมาะกับ:** แผนกที่หัวหน้ารู้จำนวนเงินอยู่แล้ว (เช่น Admin Online ที่กำหนดเรท ต่อ live)

**Flow:**
```
ใส่จำนวนเงิน → จบ (ไม่ต้องประเมินหัวข้อ ไม่มีเกรด)
```

**ฟิลด์:**
- จำนวนเงิน (required)
- หมายเหตุ (optional — บอกที่มาของจำนวน)

**ตัวอย่าง:**
```
Boat = 600 บาท
Pat = 900 บาท
weerada = 1,600 บาท
```

### 🏆 Mode C: เกรด→เงิน (`grade_incentive`)

**เหมาะกับ:** คนที่ไม่มีฐาน KPI แต่ต้องการประเมินด้วยเกรด

**Flow:**
```
หัวข้อ → คะแนน → น้ำหนัก → เกรด → ตารางมาตรฐาน → เงินรางวัล
```

**ตารางเกณฑ์:**

| คะแนน | เกรด | เงิน |
|---|---|---|
| ≥ 90 | A | 5,000 |
| 80-89 | B | 4,000 |
| 65-79 | C | 3,000 |
| < 65 | D | 2,000 |

**ตัวอย่าง:** ได้คะแนน 85 → B → ระบบให้ 4,000 บาทอัตโนมัติ

**Optional:** เพิ่ม "ค่าผลงานพิเศษ" บวกเพิ่มได้

## 4. คัดลอกแม่แบบจากการประเมินเก่า

> "เริ่มจากแม่แบบ" — feature ช่วยให้หัวหน้าไม่ต้องสร้างหัวข้อใหม่ทุกเดือน

### วิธีใช้

1. เข้าหน้าฟอร์มประเมิน `/manager/kpi/[employee_id]`
2. กดปุ่ม **✨ "เริ่มจากแม่แบบ — คัดลอกจาก KPI ที่ผ่านมา"**
3. เลือก source จาก 2 tab:
   - 📋 **คนนี้ก่อนหน้า** — เดือนก่อนๆ ของพนักงานคนเดียวกัน
   - 👥 **ลูกน้องของฉัน** — เพื่อนในทีม (ใครเคยประเมินก็ได้)
4. (Optional) ติ๊ก:
   - ☐ คัดลอก comment ของแต่ละหัวข้อ
   - ☐ คัดลอก ความเห็นภาพรวม
5. กด **"นำมาใช้"** → ฟอร์มเด้งหัวข้อ + น้ำหนักมาให้

### ที่คัดลอก vs ไม่คัดลอก

| ✅ คัดลอกอัตโนมัติ | ❌ ไม่คัดลอก (reset) |
|---|---|
| หัวข้อ (category) | คะแนน (เริ่มที่ 0) |
| คำอธิบาย | comment (opt-in) |
| น้ำหนัก (%) | ความเห็นภาพรวม (opt-in) |
| ลำดับ | |
| Mandatory flag | |

### ข้อควรระวัง

- ถ้าฟอร์มปัจจุบันมีข้อมูลอยู่แล้ว → ระบบเตือน "จะถูกทับ" ต้องกดยืนยัน 2 ครั้ง
- ใช้ได้กับ Mode A และ Mode C (Mode B ไม่ต้องมีหัวข้อ ไม่จำเป็น)

## 5. โครงสร้างหัวหน้า (Manager Chain)

ระบบรองรับ **chain หัวหน้าระดับสูง** จำกัด 1 ระดับ

### ภาพประกอบ

```
   CEO
    │
    └─ A (หัวหน้าใหญ่)
        │
        └─ B (หัวหน้าตรงของ C)
            │
            └─ C (พนักงาน)
```

- **B** = หัวหน้าตรง (Lv 1) → ประเมิน C ได้ปกติ
- **A** = หัวหน้าระดับสูง (Lv 2, skip-level) → ประเมิน C ได้ + บันทึก role = `skip_level`
- **CEO** = Lv 3 → ❌ **ไม่สามารถ** ประเมิน C (เกิน 1 ระดับ)

### ทำไมจำกัด 1 ระดับ?

- ป้องกัน CEO เห็นพนักงานเป็นร้อยคน
- ตัวอย่าง CHEN JINBIAO: chain = 175 คน → จำกัดเหลือ 30 คน (manageable)
- ยังครอบคลุม 71% ของหัวหน้าในระบบ (36 จาก 51 คน)

### Behavior

| สถานการณ์ | B (direct) ทำได้ | A (skip) ทำได้ |
|---|---|---|
| ยังไม่มีการประเมิน | ✓ | ✓ (จะถูกบันทึก `evaluator_role = skip_level`) |
| B มี draft ค้าง | ✓ | ✓ (กลายเป็น takeover — แทนที่ draft ของ B) |
| B ส่งแล้ว (submitted) | ✗ ต้อง HR reject ก่อน | ✗ ต้อง HR reject ก่อน |
| HR อนุมัติแล้ว | ✗ ใครก็แก้ไม่ได้ | ✗ ใครก็แก้ไม่ได้ |

## 6. ผู้ประเมินเพิ่มเติม (Additional Evaluators)

> สำหรับเคสที่หัวหน้าตรงไม่ใช่คนที่เข้าใจงานของพนักงานคนนี้ที่สุด เช่น พนักงานทำงานข้ามแผนก

### วิธีตั้ง (HR/Admin)

1. เข้า `/admin/employees/[id]`
2. tab **"การจ้างงาน"**
3. scroll หา section **"ผู้ประเมินเพิ่มเติม"**
4. กดปุ่ม **+ เพิ่ม**
5. ค้นหา → เลือกพนักงาน
6. เลือก scope:

| Scope | สิทธิ์ |
|---|---|
| 🟪 **KPI** | ประเมิน KPI ของคนนี้ได้ |
| 🟥 **ทดลองงาน** | ประเมินทดลองงานได้ |
| 🟦 **ทั้งหมด** | ประเมินได้ทั้ง KPI และทดลองงาน |
| ⚪ **ดูเท่านั้น** | เห็นได้ — ประเมินไม่ได้ |

7. (Optional) ใส่หมายเหตุ
8. กด **บันทึก**

### Behavior

- เพิ่มได้หลายคนต่อ 1 พนักงาน
- เห็นใน manager หน้า list ในหมวด **"ผู้ที่กำหนดให้ฉันประเมิน"**
- `evaluator_role = 'additional'` เมื่อบันทึก
- HR เห็น badge "เพิ่มเติม" บนฟอร์ม

### ลบ

กดไอคอน ❌ ข้างชื่อในรายการ → confirm → ลบ (มี audit log)

## 7. คู่มือใช้งานสำหรับ "หัวหน้า"

### 7.1 เข้าหน้า KPI

ไปที่ **เมนู KPI** ในแถบล่าง (มือถือ) หรือ sidebar (desktop)

URL: `/manager/kpi`

### 7.2 หน้า list

แสดง 3 sections:

```
┌──────────────────────────────────┐
│ 🟢 ทีมตรงของฉัน (8)              │ ← Default expand
│   • ลูก 1                         │
│   • ลูก 2                         │
├──────────────────────────────────┤
│ 🟪 ผู้ที่กำหนดให้ฉันประเมิน (2)   │ ← Default expand
│   • พนง. cross-team               │
├──────────────────────────────────┤
│ 🟦 ทีมในสาย (27)         ▼      │ ← Default collapse
│   • ลูกของลูกน้องคุณ              │
└──────────────────────────────────┘
```

แต่ละการ์ดแสดง:
- รูป + ชื่อ + ตำแหน่ง
- หัวหน้าตรง (ถ้าเป็น skip — เห็น "หัวหน้าตรง: B")
- คะแนน + เกรด (ถ้าประเมินแล้ว)
- **ผู้ประเมิน + เวลา** (ใครทำ + เมื่อไหร่)
- สถานะ: ยังไม่ประเมิน / ร่าง / รอ HR / อนุมัติ / ส่งคืน

### 7.3 ประเมินครั้งแรก

1. กดที่การ์ดของลูกน้อง
2. ที่ด้านบนเลือก **"ประเภทการประเมิน"** (Mode A/B/C)
3. ถ้า Mode A หรือ C:
   - กด **✨ เริ่มจากแม่แบบ** (ถ้ามีของเดือนก่อน)
   - หรือเพิ่มหัวข้อเอง (กดปุ่ม **+ เพิ่มหัวข้อประเมิน**)
   - ใส่น้ำหนัก (รวมต้อง = 100%)
   - ใส่คะแนน 1-100 ต่อหัวข้อ
4. ถ้า Mode B:
   - กรอกจำนวนเงิน
   - หมายเหตุ (optional)
5. (Optional) ใส่ค่าผลงานพิเศษ (bonus) + เหตุผล
6. ใส่ความเห็นภาพรวม (optional)
7. กด:
   - **บันทึกร่าง** → เก็บไว้ ไม่ส่ง
   - **ส่งประเมิน** → ส่งให้ HR ตรวจสอบ

### 7.4 แก้ฟอร์มหลังส่ง

- ถ้า status = `submitted` → กดปุ่ม **"แก้ไข"** ที่ banner สีส้ม → แก้ได้ → ส่งใหม่
- ถ้า status = `rejected` → จะเห็นเหตุผลส่งคืน → แก้ → ส่งใหม่
- ถ้า status = `approved` → แก้ไม่ได้ ต้องขอให้ HR reject ก่อน

### 7.5 Takeover (สำหรับ skip-level)

ถ้าคุณเป็น A และ B (หัวหน้าตรง) ค้างประเมินนาน:

1. ไปที่ section **"ทีมในสาย"** (expand)
2. กดเข้าฟอร์มของลูกน้องในสาย
3. ระบบจะแสดง "หัวหน้าตรง: B"
4. ประเมิน → ส่งประเมิน
5. ระบบบันทึก `evaluator_role = skip_level` + แจ้งเตือน B

## 8. คู่มือใช้งานสำหรับ "HR / Admin"

### 8.1 หน้า list ทั้งบริษัท

URL: `/admin/kpi`

แสดงรายการฟอร์ม KPI ทั้งหมด พร้อม filter:
- เดือน / ปี
- แผนก
- เกรด (A/B/C/D)
- สถานะ (submitted/approved/rejected/draft)
- ผู้ประเมิน
- คำค้นชื่อ/รหัส

### 8.2 ข้อมูลในแต่ละแถว

```
┌─────────────────────────────────────────────────────────┐
│ ชื่อพนง.    แผนก    เดือน  คะแนน  เกรด  ผู้ประเมิน     │
│ ภูมิ        บัญชี    เม.ย.  92%    A    Doris [direct] │
│                                       หัวหน้าตรง: วชิ⚠️ │
│                                       12 พ.ค. 14:32 น. │
└─────────────────────────────────────────────────────────┘
```

> ⚠️ "หัวหน้าตรง" จะแสดงเฉพาะกรณี **ต่างจากผู้ประเมิน** (เผื่อ HR ตรวจสอบความผิดปกติ)

### 8.3 Expand ดูรายละเอียด

กดที่ไอคอน ⬇ จะเห็น:

**(A) สายผู้ประเมิน:**
- ประเมินโดย [ชื่อ] [badge: role]
- ส่งประเมินเมื่อ
- HR อนุมัติเมื่อ
- ⚠️ "หัวหน้าตรงปัจจุบัน: X — ไม่ใช่คนเดียวกับผู้ประเมิน" (ถ้ามี)

**(B) เงินรางวัล KPI:**
- จำนวนเงินตามเกรด / ที่หัวหน้าใส่เอง
- ค่าผลงานพิเศษ (ถ้ามี)
- หมายเหตุ

**(C) หัวข้อประเมิน:** (Mode A และ C)
- ลำดับ + ชื่อหัวข้อ + คำอธิบาย
- น้ำหนัก + คะแนน + คะแนนถ่วงน้ำหนัก
- comment ของหัวหน้า

**(D) ความเห็นภาพรวม**

**(E) ปุ่ม Action (ถ้า status = submitted):**
- ✅ **อนุมัติ** → status เป็น approved
- ❌ **ส่งคืน** → ใส่เหตุผล → กลับเป็น rejected
- ✏️ **แก้ไข** → เปิดฟอร์ม manager เพื่อแก้

### 8.4 Bulk approve

ถ้ามีหลายฟอร์มรอ approve:
- เลือก checkbox หลายแถว
- กด **"อนุมัติทั้งหมด"**

### 8.5 Stats Dashboard

- รอ HR / อนุมัติแล้ว / ส่งคืน / ร่าง

### 8.6 หน้า Employee — ตรวจ chain

## 11. เคสตัวอย่างจริง

### เคส 1: หัวหน้าตรงประเมินตามปกติ

**สถานการณ์:**
- B เป็นหัวหน้าของ C
- B ประเมิน C เดือน เม.ย. 2026

**ขั้นตอน:**
1. B เข้า `/manager/kpi`
2. เลือก พ.ค. 2026
3. เห็น C ใน "ทีมตรงของฉัน" → กดเข้า
4. เลือก Mode A (มาตรฐาน)
5. กด **✨ เริ่มจากแม่แบบ** → เลือก "C มี.ค. 2026" → นำมาใช้
6. ปรับคะแนนต่อหัวข้อ
7. กด **ส่งประเมิน**

**ผลลัพธ์:**
- ฟอร์มสร้างใหม่ status = `submitted`, `evaluator_role = 'direct_manager'`
- HR ได้รับ notification
- C เห็นผลเมื่อ HR อนุมัติ

### เคส 2: หัวหน้าระดับสูง (skip-level) ประเมินแทน

**สถานการณ์:**
- A → B → C
- B ลาป่วยนาน ไม่ประเมิน
- A อยากประเมิน C แทน

**ขั้นตอน:**
1. A เข้า `/manager/kpi`
2. เลื่อนลง section **"ทีมในสาย"** → กด expand
3. เห็น C พร้อม badge "หัวหน้าตรง: B" → กดเข้า
4. ประเมิน → กด **ส่งประเมิน**

**ผลลัพธ์:**
- `evaluator_role = 'skip_level'`
- ฟอร์มแสดง badge "ระดับสูง" ในหน้า HR
- HR เห็น "ผู้ประเมิน: A" + "หัวหน้าตรง: B" → รู้ทันทีว่าเป็น takeover

### เคส 3: Money only — แอดมินออนไลน์

**สถานการณ์:**
- แผนก Admin Online ใช้ commission-based
- หัวหน้ารู้ตัวเลข Boat = 600, Pat = 900 อยู่แล้ว

**ขั้นตอน:**
1. หัวหน้าแอดมินเข้าฟอร์มของ Boat
2. เลือก Mode **💰 ใส่เงินเอง**
3. ใส่: **600**
4. หมายเหตุ (optional): "เรท live เดือนนี้"
5. กด **ส่งประเมิน**

**ผลลัพธ์:**
- ไม่มีหัวข้อ ไม่มีเกรด
- payroll: `bonus = 600`, `kpi_grade = 'manual'`, chip "฿ ใส่เอง"

### เคส 4: เกรด → เงิน (คนไม่มีฐาน KPI)

**สถานการณ์:**
- พนักงานใหม่ในแผนกที่ไม่มี kpi_bonus_settings
- ใช้เกณฑ์มาตรฐาน 5,000/4,000/3,000/2,000

**ขั้นตอน:**
1. หัวหน้าเข้าฟอร์ม
2. เลือก Mode **🏆 เกรด→เงิน**
3. เห็นตารางเกณฑ์เด้งขึ้นมา + คะแนนปัจจุบันจะ highlight cell
4. ประเมินหัวข้อ → ได้คะแนน 87 → เกรด B
5. ระบบขึ้น **"จะได้ 4,000 บาท"** ทันที
6. (Optional) ใส่ค่าผลงานพิเศษ 500 + เหตุผล "เกินเป้า"
7. กด **ส่งประเมิน**

**ผลลัพธ์:**
- เกรด B, incentive = 4,000, bonus = 500
- payroll: bonus = 4,500, kpi_grade = 'B'

### เคส 5: ค่าผลงานพิเศษ (bonus) บนงานปกติ

**สถานการณ์:**
- พนักงานทำงานพิเศษเสริม
- ปกติประเมินด้วยฐาน KPI 5,000 บาท
- เกรด A → ฐาน 5,000 × 1.2 = 6,000
- เพิ่ม bonus 1,000 → รวม 7,000

**ขั้นตอน:**
1. ประเมินตามปกติ (Mode A) → เกรด A
2. ในช่อง "ค่าผลงานพิเศษ" ใส่ **1,000**
3. ใส่เหตุผล: "ช่วยทำโปรเจคต์ X ในเดือนนี้"
4. กด **ส่งประเมิน**

**ผลลัพธ์:**
- payroll: bonus = 7,000, kpi_grade = 'A'

### เคส 6: HR ส่งคืนให้แก้

**สถานการณ์:**
- B ประเมิน C → เกรด A
- HR เห็นว่าน้ำหนักไม่ครบ 100% → ส่งคืน

**ขั้นตอน HR:**
1. เข้า `/admin/kpi` → expand ฟอร์ม
2. กด **"ส่งคืน"** → ใส่เหตุผล: "น้ำหนักรวมเกิน 100% กรุณาแก้"

**ขั้นตอน B:**
1. ได้ notification
2. เข้าฟอร์ม → เห็น banner สีแดง "HR ส่งคืน — กรุณาแก้ไขและส่งใหม่"
3. แก้น้ำหนัก → กด **ส่งประเมิน**

**ผลลัพธ์:**
- status: `rejected` → `submitted` ใหม่
- HR ตรวจอีกรอบ

### เคส 7: Additional evaluator

**สถานการณ์:**
- พนักงาน "พล" ทำงานกับ 2 ทีม:
  - ทีม A: B เป็นหัวหน้าตรง
  - ทีม B (cross-functional): D อยากประเมินด้วย

**ขั้นตอน Admin:**
1. เข้า `/admin/employees/[พล-id]` → tab "การจ้างงาน"
2. section "ผู้ประเมินเพิ่มเติม" → กดเพิ่ม
3. ค้นหา D → เลือก
4. Scope: **KPI**
5. หมายเหตุ: "ดูงาน project X"
6. บันทึก

**ผลลัพธ์:**
- B และ D **ทั้งคู่** สามารถประเมิน "พล" ได้
- ใน `/manager/kpi` ของ D → เห็น "พล" ใน section "ผู้ที่กำหนดให้ฉันประเมิน"
- ถ้า D ประเมิน → `evaluator_role = 'additional'`

## 13. แก้ปัญหาที่พบบ่อย (FAQ)

### Q1: ลูกน้องบางคนไม่ขึ้นในหน้า KPI ของฉัน

**สาเหตุที่เป็นไปได้:**
1. ไม่ได้ตั้งความสัมพันธ์ใน `employee_manager_history` → admin ไปที่ `/admin/employees/[id]` → tab "ประวัติหัวหน้า" → ตั้งคุณเป็นหัวหน้า
2. มี `kpi_evaluator_id` override ไปคนอื่น → admin ตรวจที่ tab "การจ้างงาน" → section "ผู้ประเมิน KPI"
3. ลูกน้องอยู่คนละบริษัท → ไม่มีปัญหา (รองรับ cross-company)

### Q2: ประเมินแล้วเงินไม่ขึ้น payroll

**ตรวจ:**
1. status ต้อง `submitted` / `approved` / `acknowledged`
2. เดือน + ปีของฟอร์มตรงกับ payroll period มั้ย
3. ถ้า Mode A: มี `kpi_bonus_settings` หรือไม่ — ถ้าไม่มี เงินจะเป็น 0 (ยกเว้นมี bonus)
4. ลอง regenerate payroll ใหม่ที่ `/admin/payroll`

### Q3: อยากเปลี่ยน mode หลังประเมินแล้ว

ทำได้ — กดแก้ฟอร์ม → เลือก mode ใหม่ → คะแนน/เงินจะรีเซ็ตตาม mode → ส่งใหม่

### Q4: HR เห็น "หัวหน้าตรง: X ⚠️ ไม่ใช่คนเดียวกับผู้ประเมิน" หมายความว่าอะไร?

= หลังจากประเมินไปแล้ว มีการ **เปลี่ยนหัวหน้า** ของพนักงาน หรือเป็นการประเมินโดย skip-level / additional

ไม่ใช่บัค — เป็น flag ให้ HR ตรวจสอบ:
- ถ้าเป็น skip-level / additional → badge ในแถวจะระบุ → ไม่ต้องกังวล
- ถ้าเป็น direct_manager แต่ชื่อต่างจาก หัวหน้าตรงปัจจุบัน → อาจมีการย้ายหัวหน้าหลังประเมิน

### Q5: ตารางเงินรางวัล Mode C เปลี่ยนได้มั้ย?

ปัจจุบันเป็นค่าคงที่ใน code (`KPI_GRADE_INCENTIVE_TABLE` ใน `src/lib/utils/kpi.ts`):
- A: 5,000 / B: 4,000 / C: 3,000 / D: 2,000

ถ้าต้องการปรับ → แก้ใน code + deploy ใหม่

อนาคต: อาจทำให้ตั้งใน Admin → company settings ได้

### Q6: Skip-level เห็นได้กี่ระดับ?

ปัจจุบัน **1 ระดับ** (`SKIP_DEPTH = 1` ใน `src/lib/utils/evaluator-chain.ts`)

แปลว่า: A → B → C → D — A เห็น C (skip-1) ได้ แต่ไม่เห็น D (skip-2)

ถ้าอยากเปิดให้ลึกขึ้น → แก้ค่า `SKIP_DEPTH` ใน code

### Q7: ลบหัวหน้าเพิ่มเติมแล้ว ฟอร์มเก่าที่เขาประเมินไปแล้วจะเป็นอะไร?

ฟอร์มเก่ายัง **อยู่ปกติ** — `evaluator_id` ยังชี้ไปคนเดิม, `evaluator_role` ยังเป็น 'additional'

แค่ไม่สามารถประเมินฟอร์มใหม่ของพนักงานคนนี้ได้แล้ว

### Q8: Bonus ใส่ได้สูงสุดเท่าไหร่?

**ไม่จำกัด** (เป็นเลขทศนิยม) — admin ควรกำหนด guideline เอง

### Q9: HR แก้คะแนนของหัวหน้าได้มั้ย?

ได้ — HR กดปุ่ม **"แก้ไข"** ในหน้า expand → จะเด้งไปฟอร์ม manager → แก้ → กด "ส่งประเมิน" → ฟอร์มเป็น `approved` ทันที (ไม่ต้องผ่าน HR เพราะเป็น HR เอง)

### Q10: ทำไมประเมินตัวเองไม่ได้?

ป้องกันการเช็คอินตัวเอง — `canEvaluate()` ตรวจว่า evaluator ≠ employee
